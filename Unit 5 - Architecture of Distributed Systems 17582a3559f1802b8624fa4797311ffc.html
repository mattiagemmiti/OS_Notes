Unit 5 - Architecture of Distributed Systems

5.1. Client-Server-Systems and Distributed Applications

  • Early Monolithic Programs to Modular Systems:

    Monolithic Programs: In the past, software was simple and combined all functionality into a single component. This approach worked when program requirements rarely changed and complexity was low.

    Modular and Multi-Layered Architecture: As software grew in complexity and a greater avaliability of resources, modular and multi-layered designs emerged, dividing software into distinct layers with specific responsibilities. Layers interact top-down via defined interfaces, reducing the complexity within the layers, avoiding dependency cycles (a function call across layers may only ever be made from top to bottom) and allowing for interchangeability of individual layers without having to change the structure of the overall system. Furthermore, the encapsulation in layers improves maintainability, scalability, and expandability. Today, this multi-layered approach is standard and a basic requirement for distributed applications.

  • Two-tier Architecture vs Three-Tier Architecture:
    1. Two-Tier (or Client-Server) Architecture:

      Comprises two layers where the server offers functions, while the client request services:

      • Client (Upper Layer): Service user, that invokes the functions of the lower layer.
      • Server (Lower Layer): Service provider.

      These layers may reside on the same or separate computers. When on the same computer, the two layers could be embodied by two software modules.

    1. Three-Tier Architecture:

      Adds a third layer to the two-layer architecture and divides responsibilities:

      • Presentation Tier: User interface. Responsible for visual presentation and input handling.
      • Logic Tier: Handles data processing mechanisms of the application (also called application or business logic tier).
      • Data Tier: Manages and stores data (e.g., manages data records in databases).

      These layers can be distributed across multiple machines, allowing better load balancing and scalability (multiple clients can interact with servers distributed across various locations.).

  • Thin vs. Fat Client Approaches
    1. Thin Client:

    Only the presentation tier runs on one or many client computers, while Logic and data processing occur on the server. This makes it suitable for scalability, as servers handle most of the workload.

    1. Fat Client:

    The client computers also process the data (Logic tier components), and the server only provides persistent storage for these processed data. This results in less dependence on servers but increased use of client resources.

    Further Approaches:

    Because the different tiers are logically separated from each other, further approaches
    using different distribution strategies can be realized, such as a distributed architecture according to the three-tier model, where the accesses of several clients are mapped individually (e.g., by load balancing) to different application servers. These, in turn, access a common, distributed database.

    • Load balancing= Distribution of large amounts of data, large computing loads,
      and/or many processing requests to several parallel systems.
    • Example of a Distributed System on a Client-Server Basis
  • Horizontal vs Vertical Distribution

    In the distributed Applications, two distributions can be generally distinguished:

    Horizontal Distribution:

    • Redundant servers handle client requests, enabling load sharing, because each server can answer the request itself.
    • Scaling involves adding more servers (of the same nature) to handle increased client requests (of the same nature).
    • Horizontal scaling ensures that no single server is overwhelmed, and load balancing across multiple servers further improves efficiency and fault tolerance.

    Vertical Distribution:

    • Different, specialized servers handle different tasks/functions/data. Therefore, a client with a certain request can only address a certain server (e.g., one server handles the application layer, and another handles the database layer).
    • Scaling: the more client requests (of the same nature), the more powerful the respective server should be (increasing the computing power, memory, or storage of each server)

    Combining both scaling approaches ensures that systems can meet growing demand without compromising performance.

  • Distributed Applications:

    Software systems where components are distributed across different computers, meaning they constantly exchange data with each other via networks. This distribution allows for greater scalability, fault tolerance, and performance optimization.

5.2. Communication, Synchronization, and Transactions

Communication in Distributed Systems

Communication in distributed systems typically follows the client-server model, where a client requests specific resources or services from a server. Distributed systems use internet protocols like HTTP but often require additional protocols for complex interactions (e.g., server-to-server or peer-to-peer communication). This complexity goes beyond simple client-server models, such as retrieving web pages.

  • Remote Procedure Calls (RPCs):

    Remote Procedure Calls (RPCs) are a protocol that enables a process on one computer to execute a function or service as if it were local, even though the service is located on a remote server. The client sends a request (similar to a function call) and receives a response in the form of a result or return value.

    RPCs were introduced early in network development to make remote function calls more transparent. They bridge the gap between local and remote function execution, ensuring seamless interaction for the client.

    Functions vs. Procedures:

    A function is generally stateless, meaning it doesn't rely on or modify any internal state.

    In contrast, a procedure typically operates on mutable data within its internal state. In the RPC model, the server provides the procedure, and the client invokes it over the network. To simplify this process for the client, an abstraction layer ensures the remote procedure feels like a local procedure call. On the client side, this layer includes client stub procedures, which act as local replicas of the actual service procedures on the server. While these stubs don’t execute the service themselves, they simulate the remote procedures locally, forward the requests to the server and return the result to the client. This design makes communication in distributed systems seamless, with HTTP often used as the transport protocol, enabling efficient remote access to services and resources.

    • Schematic of an RPC with Client-Side Perspective
    • Communication Modes:

      In remote procedure calls (RPC), there are two primary modes of communication for handling client-server interactions: active waiting (polling) and passive waiting (interrupt-driven). These modes determine how the client waits for the server’s response and how system resources, like the CPU, are used.

      • active waiting with polling: The client continuously checks (polls) for the server’s response. This keeps the CPU busy, as the client remains active and engaged in the RPC process.
      • Passive Waiting (Interrupt-Driven): The client passes control of the RPC to the operating system and goes to sleep until the server responds. The CPU is freed for other tasks during this waiting period. While the response may take longer due to the overhead of waking the client back up, this method is often preferred for network-based RPCs, as it avoids constant CPU usage.

      If the results of the RPC are not needed immediately (e.g., because the client sends the RPC in the program code early and with foresight), the client itself could continue calculating until it reaches a point where that RPC result is needed. To realize such an overlapping of further computation and communication in the background, many systems offer non-blocking RPC variants in addition to the standard blocking ones:

      • Blocking Communication: The client waits within the RPC for the server’s response, either actively (polling) or passively (interruption), and cannot perform other tasks in the meantime.
      • Non-blocking Communication: The client continues processing other tasks while waiting for the server’s response, enabling concurrency or parallelism between server and client operations. A reference to the call running in the background is then returned in the form of a handle. With this handle, the client can check to see if the server result is available in the meantime (active polling).

Synchronization Methods

In computing systems, whether local or distributed, parallel and/or asynchronous processes must still be able to coordinate and thus synchronize with each other at some point. This synchronization is usually achieved through system calls provided by the local operating system for processes running together on the same computer.

  • Local Synchronization Methods:

    Mutex (Mutual Exclusion): In a system with multiple processes, only one process at a time is allowed to perform a specific operation or use an exclusive resource. This is managed using a token-based system, where a process must hold the token (called a mutex) to gain access to the resource. Other processes must wait until the token is released.

    • Purpose: Ensures that critical resources or operations are not accessed simultaneously, preventing conflicts or inconsistencies.

    Semaphores (Atomic Counters variables): variables used for synchronization, allowing up to n processes to share a resource simultaneously. If the number of simultaneous users exceeds n, additional processes must wait until the resource becomes available. There are 2 types of Semaphore:

    • Binary Semaphore (n=1): Works like a mutex, allowing only one process at a time.
    • General Semaphore: Allows multiple processes (up to n) to access the resource.

    Barrier: A synchronization mechanism that ensures all n processes reach a specific point before any of them can continue. When a process arrives at the barrier, it must wait until exactly n processes have arrived. Only then are all processes allowed to proceed, ensuring they are synchronized at a specific stage and enabling coordinated actions across the system.

  • Distributed Synchronization:

    Unlike local systems, distributed systems cannot rely on direct system calls for synchronization, as each computer has its own operating system. Instead, communication happens via network messages, according to a synchronization protocol. The mechanism is:

    1. Token Passing for Mutex: A special token is passed between processes (or computers). A process must hold the token to gain access to the critical section. To acquire the token, a process sends a request — either to all other processes or, depending on the method used, to a specific process that currently holds the token (or is known to have held it previously).
    1. Distributed Barriers: Processes form a logical ring, with each having a predecessor and a successor. A token circulates twice around the ring to ensure all processes have synchronized. Once this is understood, every process can leave the barrier function. In distributed systems, synchronisation ensures that multiple processes coordinate effectively.

      Distributed Barrier with Token Passing

    Challenges: Synchronization in distributed systems can fail if a process or computer crashes or a message containing the token is lost. Because there is usually no central observer as in the local operating system, such a failure can hardly be detected globally.

    The situation where processes indefinitely wait for a token or message that will never arrive, often due to failure, is called a deadlock. Deadlocks are typically addressed using timeouts, where processes stop waiting if no response is received within a specific time.

Transactions

In distributed client-server systems, persistent data storage is critical for maintaining consistent and reliable access to shared data. The challenges arise when multiple servers interact with the same dataset, potentially leading to inconsistencies due to differences in data views or processing orders if proper protocols are not in place. The solution is to use Transactions to ensure data consistency and reliability during access and modification.

A transaction is a sequence of operations that ensures reliable and traceable modifications of persistent data on a storage system, particularly when the system could be affected by failures. The goal is to eliminate uncertainty about the state of data during and after operations.

  • They follow the ACID properties:
    • Atomicity: A transaction is either fully completed or not executed at all (If a payment fails, the flight reservation is also canceled (rollback)).
    • Consistency: The database remains in a valid state before and after a transaction. No intermediate or invalid states are visible externally.
    • Isolation: Simultaneous transactions do not interfere with each other through their overlapping. This means that write accesses in particular must be serialized.
    • Durability: Once a transaction is committed, its effects are permanent, even if the system fails afterward.
  • Transaction code

    A transaction meets these criteria by encapsulating a set of (data changing) operations in
    such a way that they are either ordered and executed entirely or not at all. The set of operations that form a transaction begins with a
    begin function and include a series of operations (read/write).

    • If successful, they end with a commit function, making the changes permanent.
    • If there’s a failure, a rollback reverts the system to its initial state before the transaction began. This is necessary when a transaction cannot be completed successfully due to errors or interruptions.
  • Distributed Transactions

    In distributed systems, transactions span multiple servers or databases. Failures in any component (server, connection to the client or other servers, or transaction errors) can cause inconsistencies.

    Solving the issue of processing distributed transactions across multiple servers require global coordination. A coordinator ensures that either all interdependent (sub-) transactions are successful or that a global rollback is performed.

    Example Workflow:

    • A client sends a transaction to a coordinator, because the global coordination can be done by the client or by an intermediate server
    • The coordinator splits the transaction into sub-transactions for different databases, following a protocol that ensures all sub-transactions are either completed or globally rolled back.

    Challenges like deadlocks and process failures can occur, requiring careful management.

5.3. Service Orientation and Cloud Applications

In addition to the classic client-server model, extended architectural approaches like Service-Oriented Architectures (SOAs) and Microservices offer more flexibility, modularity, and interoperability in distributed systems. These architectures focus on how services interact and are orchestrated to meet both technical and business needs.

  • Service-oriented Architecture (SOA)

    SOA is an architecture where systems are composed of independent, loosely coupled services that can be orchestrated very flexibly and communicate using messages. Key features are:

    Service Offering: Servers provide and advertise their services through global directories. Global directories include a description of each API function, such as those described using OpenAPI, and details of available services and how to use them.

    Interoperability: Services are platform-independent, enabling seamless integration across different systems.

    Orchestration: Higher-level services are created by combining subordinate services for flexible and reusable solutions.

    • Schematic of a Service-Oriented Architecture (SOA)

      A service directory could have as components:

      • Service User: The client requesting a service.
      • Service Provider: The server offering the service.
      • Service Broker: An intermediary maintaining a directory of services.

    Business Alignment: SOA bridges IT architectures with an organization’s business processes, ensuring IT services align with business needs.

  • Microservice Architecture

    Microservices build upon SOA but focus on smaller, more fine-grained services. They are an extension of the SOA concept. Key features are:

    Lightweight Services: Each service performs a single, well-defined function, making the system easier to develop, test, and maintain.

    Fine-Grained Modularity: Each service has a clear interface, simplifying interactions.

    Fault Tolerance and Parallelism: Redundant service instances improve reliability and allow parallel processing.

    Lightweight Protocols: Communication between services uses simple and efficient protocols, reducing complexity.

  • Web Services

    Web services provide a middleware layer that enables users to have uniform access to remote and distributed services, often based on the Service-Oriented Architecture (SOA). These services must also offer a high degree of portability with flexible standards. Key features are:

    Middleware: Software layers that provide communication, coordination, and service brokering beyond what the operating system offers. In the OSI model, middleware typically operates around the fifth layer (Session) and the seventh layer (Application).

    Protocols and Standards: The communication for accessing these services is usually done via well-known web service protocols and standards such as:

    • Simple Object Access Protocol (SOAP): An HTTP-based protocol for sending RPCs.
    • Extendable Markup Language (XML): Encodes SOAP messages for interoperability.
    • Remote Procedure Calls (RPCs): Facilitate function-like service calls over networks.

    Stateless Processing: Services do not maintain internal states and therefore always returns the same result for the same call with the same parameters. A service user or client can therefore contact any server as long as it offers the same web service. To find out which server provides which service, clients can contact directory services.

    Because, for a web service to operate in a stateless manner, all required information must always be supplied by the client in the form of parameters, the Representational State Transfer (REST) paradigm has been established, and Web services following this paradigm are called RESTful.

    WSDL: A standard for describing web services in a portable way, specifying inputs, outputs, and parameters.

  • Cloud Computing

    Cloud computing refers to the processing and storage of data on remote servers (the "cloud"), making computing resources available on-demand over the internet.

    • Cloud Architectures

      Software as a Service (SaaS): For end users: Provides fully developed applications that address the human user and web application as the end customer. Data processing and collaboration occur in the cloud. Examples include Marketplaces (eBay, Amazon), Online software suits (Microsoft Office 365), and website builders (Jimdo).

      Platform as a Service (PaaS): For Cloud Application Developers: Supports the software product cycle for web application, offering tools and platforms to build, deploy, and manage applications. Examples include Google App Engine, Microsoft Azure, and data service platforms, which provide weather forecasts or stock exchange data for further processing

      Infrastructure as a Service (IaaS): For IT Administrators: Provides direct access (and use of) to abstract cloud infrastructure resources like virtual servers, storage, and networking. IaaS uses virtualization to abstract hardware, enabling elastic scaling (dynamic allocation of resources), and protecting the physical hardware from the customer. Examples include Amazon Elastic Compute Cloud (EC2) and DigitalOcean.

      • Layer Models of Cloud Computing Architectures
    • Cloud infrastructure

      Where the services offered are eventually implemented is irrelevant in cloud computing. These services are often located in dedicated large-scale data centers used by cloud service providers. The data centers are usually structured according to the principle of server farms.

      • Virtualization: Optimizes hardware usage by running multiple virtual instances on each physical server. This helps the system react very flexibly to scale horizontally (adding/removing virtual servers dynamically) for handling varying loads.
      • Transparency: Users don’t know or need to know the exact location of the data centers.

5.4. Computer Clusters

  • Cloud Computing and Server farms

    Cloud computing relies on server farms, aggregations of many similarly configured virtual servers hosted in large-scale data centers, that work close to one another. This allows for a very flexible (i.e., elastic) load balancing between the server instances. These servers provide services to a wide variety of clients. Communication is outward-facing, primarily directed toward clients using the services. Data centers are interconnected but prioritize communication with external clients over internal interactions.

  • HPC Systems:

    These are specialized data centers designed for scientific research and supercomputing.

    HPC typically comprises many high-performance servers, which differ from cloud computing:

    • Inward-Focused Communication: Servers work together to solve a single, complex computing task.
    • Task Division: Large tasks are split into smaller subtasks, which are processed in parallel by multiple servers. Each computer is assigned only its share
    • Intermediate Results Exchange: Servers frequently communicate intermediate results to ensure the subtasks align because the subtasks are typically similar and are rarely independent of each other.

    Communication between servers follows mostly homogeneous communication patterns between the computers, and uses high-performance networks for fast data exchange, especially between neighboring servers.

    The hardware infrastructure and network topology of HCP is referred to as a computer cluster or a distributed parallel computer, and it’s designed for highly efficient, synchronized processing.

  • Parallel Processing

    Tasks are divided into subtasks and assigned to multiple computers. Each computer processes its subtask, exchanging results with others for subsequent steps.

    • Example: Weather Forecasting:

      The atmosphere is divided into volume elements, with each computer responsible for simulating one element. Each computer calculates the state of the flow within its assigned volume element for a very short time step into the future. Afterward, they exchange results over the network with neighboring computers, which simulate adjacent volume elements. These updated values are then used to calculate the next time step. By repeating this process, the overall airflow and weather forecast can be gradually computed and predicted.

    Key Trade-offs are that increasing the number of computers enhances accuracy but adds communication overhead (the calculation of a single simulation in scientific computing centers can take up several days). So, the more computers in use means, the more will be the communication effort, which could negatively impact the time needed to obtain a result: the high-performance networks can no longer compensate for this increased activity.

    This trade-off between runtime, accuracy, and the use of computational resources is called parallel scaling, which has two variants:

    • Weak Scaling: Adds more computers to improve result accuracy with constant runtime.
    • Strong Scaling: Adds more computers to reduce the time it takes to achieve a result, with a focus on maintaining result accuracy. Here, we speak of temporal acceleration (the speedup, the metric describes by which factor the processing is accelerated in the parallel case compared to the sequential case).
      • Speedup (Sₙ): Measures how much faster n computers in a network are compared to one. This speedup is simply calculated from the time T(1), which is a single computer, related to the time T(n), which are n computers, needed for the same result. This is noted as:
      • Ideal speedup is if the result were avaliable n-times faster when using n computer, but communication overhead usually reduces this in practice.
  • Parallel Programming

    The client-server model commonly used for distributed applications isn't well-suited for clusters because tasks in a cluster are similar and need to work in parallel, unlike the diverse roles of clients and servers.

    1. The Message-Passing Interface (MPI) is designed specifically for communication between the computers of a cluster. It facilitates the exchange of messages between processes running in parallel on different computers.
    1. MPI standard also covers session management, aspects of data representation, and the storage of data via input/output functions in such systems.
    1. In contrast to the client-server model, in which client and server are typically processing completely different programs, MPI follows a single-program-multiple-data paradigm (SPMD) where a single program at a time is written and deployed across all computers in the cluster. Each process in the cluster runs the same program but operates on different portions of the data. The processes that run this single program in parallel can pass messages to each other directly over the connecting network (e.g., for exchanging any intermediate results). It is this message passing to which the standard owes its name. This eliminates the need to rely on IP addresses or port numbers. Instead, MPI assigns each process a unique rank (number in ascending order starting from zero and running up to the number that reflects the exact number of processes of the respective session) to identify and address it.
    1. The MPI standard provides a comprehensive set of over 400 functions designed to support communication and synchronization in distributed systems, particularly for parallel computing. Examples include blocking and non-blocking communication functions, the realization of collective communication patterns (broadcasting), and synchronization functions (e.g., the barrier function).
    1. Only transactions are foreign to MPI. MPI is an open standard according to which several
      communication libraries have been developed as middleware for various parallel computing platforms. Well-known, open-source representatives of such libraries include Open MPI, MPICH, MVAPICH, and ParaStation MPI
