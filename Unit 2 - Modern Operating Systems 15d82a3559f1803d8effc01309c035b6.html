2.1. Microsoft Windows

Developed by Microsoft Corporation

History
  • MS-DOS: The official OS of the IBM PC, released in 1981. It was designed as a simple, single-user operating system with 16-bit address space. Initially, it had no multitasking functionality, which simplified the management of the floppy and hard disks. The only user interface offered at that time was a text-based command line interpreter
  • MS-DOS based Windows: The first versions of Windows were essentially GUI extensions of MS-DOS (Microsoft Disk Operating System). They were mere graphical window managers that still required MS-DOS for interfacing with the hardware.
  • NT-based Windows: Microsoft developed a new kernel, known as New Technology (NT), which introduced new CPU capability, better memory protection and the NTFS file system. NT also provided partial compatibility with the older DOS environment.
  • Windows Modern UI: Modern versions of Windows, starting from Windows 8, introduced the Modern UI, designed to support more type of devices, touch screen support, and tile based UI.

Structure & Interfaces

In kernel mode, the central layers of the operating system handle two main functions:

  1. Hardware Management: This includes interruption handling and hardware-related CPU management.
  1. System Services: These manage processes, threads, virtual memory, and input/output operations, supported by device drivers.

At the lowest software level, the Hardware Abstraction Layer (HAL) provides interfaces to hardware components like the interrupt controller, MMU, and device controllers. At higher levels, the Win32 API, introduced with Windows NT, offers a standardized programming interface for accessing system libraries and abstracting system calls in the Windows kernel.

The Win32 API ensures compatibility across Windows versions. Applications written using it can run on Windows NT, 95/98/ME, and subsequent versions like 2000, Vista, 7, 8/8.1, and 10, making it a widely supported interface for Windows development.

Win32 API is a programming interface that provides standardized access to system libraries and abstracts system calls in the Windows kernel. It was introduced with Windows NT and is used for programming Windows applications. Functions like CreateFile, ReadFile, and WriteFile are part of this API.

File System

Windows NT introduced the New Technology File System (NTFS), which remains the default file system for modern Windows versions, supporting large file sizes and features like journaling. NTFS replaced the older File Allocation Table (FAT) system, which was widely used for floppy disks, hard drives, and, more recently, devices like memory sticks.

  • in FAT all allocation information between all files and all blocks of a disk or drive are stored in a single, large table. FAT32 (File Allocation Table) is still used in portable devices but has limitations, such as a 4GB maximum file size.

Key differences between NTFS and FAT32:

  • File Size Limits: NTFS supports files up to 16 Tebibytes, while FAT32 is limited to 4 Gibibytes per file, making it unsuitable for large files like uncompressed DVD images.
  • Volume Size: NTFS can handle volumes up to 256 Tebibytes, compared to FAT32's 2 Tebibyte limit.
  • File Count: NTFS allows more files than FAT32.

NTFS also offers advanced features:

  • Journaling: It logs metadata changes in a journal to ensure data consistency, even after power outages or system crashes, avoiding lengthy file system checks. This allows for the consistent state of data structures to be instantly restored at any time.
  • Modern Use: NTFS is optimized for hard disks, while FAT32 remains popular for cross-platform compatibility on portable devices.

This makes NTFS the superior choice for larger volumes and critical systems requiring reliability and scalability.

2.2. Case Study: Unix and Linux

Unix History:

Unix was originally developed in the 1970s at AT&T's Bell Labs and became highly influential due to its portability and multitasking capabilities using C programming language. Unix was open-source operating system until 1980 and was commonly used at research institutes and universities.

Common early and modern derivatives (forks and hardware manufacturers) of Unix include

  • Berkeley Software Distribution (BSD)
  • Unix System V
  • SunOS/Solaris (today Oracle)
  • FreeBSD, OpenBSD, and NetBSD (open-source)
  • Mac OS X/macOS
  • Xenix (licensed by Microsoft)
Linux History:

In 1987, Tanenbaum created MINIX, an open-source teaching operating system designed for academic use. Building on MINIX, Linus Torvalds developed the Linux kernel in 1991, releasing it under the GNU General Public License (GPL). This license promoted open-source collaboration while protecting the code from misuse in proprietary projects. When combined with GNU software, Linux forms a complete operating system often referred to as GNU/Linux. It is an open-source version of Unix and quickly gained widespread adoption.

Since its release, Linux development has been driven by the online open-source community, with maintainers overseeing specific sections of the code. Linus Torvalds continues to have the final say on kernel changes. Over time, an extensive ecosystem of tools, libraries, and runtime environments has grown around Linux, leading to the creation of various Linux distributions. These distributions often share the same kernel version but may differ in configuration and packaging. Linux now supports a wide range of uses, from personal computing to enterprise systems.

  • GNU General Public License

    This license (also called GNU GPL) allows anybody to use, modify, and distribute open-source software provided that the rights outlined in the license are always reserved during redistribution and re-use.

Structure and Interfaces

Unix-like operating systems share a similar structure and interface design, based on standardized software layers with interchangeable components. This design allows for easy replacement or modification of one layer without affecting others. A clear example is the graphical user interface (GUI) and desktop environment, where users can choose between different options like GNOME or KDE in Linux-based systems. This flexibility highlights the modularity of Unix-like systems.

POSIX (Portable Operating System Interface) is a set of standardised operating system interfaces and APIs that ensure compatibility between different UNIX-like systems. It defines a common set of standards, ensuring that software written for one POSIX-compliant system can run on another without requiring major modifications. It was developed by IEEE and the Open Group (ISO/IEC 9945). POSIX defines:

  • Programming Interfaces: Standardized functions for system calls, described in C, enabling portability across compliant systems.
  • Tools and Shells: Command-line tools and interpreters (e.g., shell scripts) that work seamlessly across POSIX-compliant systems. They provide an interface for users to interact with the system, often through command-line interfaces (CLI).

This standard allows POSIX-compatible programs and scripts to run on any compliant Unix-like OS without modification. While Windows is not Unix-like and lacks full POSIX compliance, tools like the Windows Subsystem for Linux (WSL) and hypervisors enable interoperability and portability between Windows and Unix-based systems.

  • Multitasking: Unix introduced true multitasking, allowing multiple processes to run simultaneously.
  • Multiuser Support: Both Unix and Linux are designed to handle multiple users simultaneously, providing security and resource isolation between users.
  • Kernel: The core part of Unix/Linux responsible for managing resources and system calls.

File Systems:

Unix and Linux primarily use inode-based file systems (like ext4 for Linux), which efficiently manage files and directories.

"everything is a file" is a fundamental design philosophy. It means that almost all system resources, whether they are actual files, hardware devices, or even processes, are treated as files within the system.

  • In Unix/Linux systems, the special file /dev stands for “devices” and /lp for “line printer.” Writing or copying data to this special file would then cause exactly this data to be printed.

In Unix-like operating systems, mounting allows different file systems to be attached to the global directory tree under specific subdirectories. Here’s how it works:

  • Each storage device (such as a hard disk or USB drive) is treated as a block device, meaning it can be accessed similarly to a file.
  • Before you can access files or directories on a device, the system must mount its file system into the global directory structure.
    • For example, a USB drive could be mounted at /mnt/ssd, making its contents accessible as part of the system's overall directory tree.

APFS stands for Apple File System, designed for macOS and iOS.

2.3. Mobile Operating Systems

Mobile OSs are designed for devices like smartphones and tablets, where resources (battery, memory, and processing power) are limited. These operating systems must support mobility, meaning they handle frequent changes in network connectivity (Wi-Fi, cellular data) and location-based services.

Android is a popular mobile OS built on a Linux kernel. It uses the Android Runtime (ART), which allows Android apps to run in their own virtual machines with strict access control over private data, system functions, and hardware resources, isolated from the system and other apps. This provides security and stability. Android apps are written in Java.

The Android Application Framework provides access to the application layer for Android apps. The hardware resources of the mobile device can only be accessed via system libraries provided by underlying layers of Android OS. These libraries encapsulate the actual system calls in the kernel and make these functions available to the actual applications via the Android Application Framework. For instance, if an application needs to use the device's sensors, it will use the system libraries to make a call to the kernel which will in turn communicate with the hardware resource i.e., the sensor. All this routing is made possible via the Android Application Framework, making the Framework crucial for accessing and managing hardware resources in a device.

Android’s security model is built on a granular permissions system that governs how apps interact with sensitive data and system functions. Android's security concept includes running apps in separate virtual machines with strict access control over private data, system functions, and hardware resources.
